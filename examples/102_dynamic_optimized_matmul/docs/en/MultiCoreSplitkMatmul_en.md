# MultiCoreSplitkMatmul

## 1. Template Description

Due to hardware instruction restrictions, m1 and n1 must be multiples of 16 (m1 and n1 are L1Tile parameters). When Matrix C is small, for example, with a size of 16 × 16, only one basic task block can be partitioned. If K is very large at this time, the computation efficiency will be very low because the computation workload is heavy, but only one core participates in the computation, causing a serious waste of computing power and bandwidth. Furthermore, when m1 and n1 are too small, it may lead to low efficiency of movement instructions. Using larger m1 and n1 might yield better results, but larger m1 and n1 will reduce the partitioned task blocks and use fewer cores, thereby resulting in more resource waste.

Multi-core Split-K can solve the above problems. It not only partitions cores along the M and N directions, but also involves core partitioning in the K direction, distributing the computation tasks of the K direction across multiple cores. When M and N are small and K is large, it will have a great advantage compared to ordinary templates. For example, for a Matmul with M = 16, N = 16, and K = 10240, if m1 = 16, n1 = 16, and k1 = 256, and if partitioning is only performed on M and N, there can only be one task block. However, there are 40 task blocks in the K direction. Assuming the number of cores is 3, the K direction can be divided into 3 parts, with each core computing one of the parts. Finally, the Vector Core is used to accumulate the partial sums, thereby utilizing all computing power resources.

The key points of this template are explained as follows:

1. For the partitioning in the K direction, partitioning is performed at the granularity of basic blocks. For example, for a Matmul with M = 16, N = 16, and K = 10240, where m1 = 16, n1 = 16, and k1 = 256, 40 task blocks are partitioned in the K direction. Assuming the number of cores is 3, the 40 task blocks are divided into 3 parts, which are 14, 13, and 13 respectively (evenly divided; if there is a remainder, the remainder is evenly distributed to the preceding cores). At this time, Core 0 computes the K segment from 0 to 14 × 256, Core 1 computes the K segment from 14 × 256 to 27 × 256, and Core 2 computes the K segment from 27 × 256 to 40 × 256.
2. It is necessary to request workspace. The number of workspace copies to be requested is equal to the number of segments the K direction is divided into. The size of each workspace is `C_SIZE × sizeof(ElementAccumulator)`. The required workspace is relatively large, so this template is not suitable for scenarios with a large Matrix C.
3. After the Cube completes the computation of partial sums, a full-core synchronization is required to ensure that all partial sums are computed. At this time, the Vector can begin the accumulation operation. To ensure the determinism of the computation results, the Vector will perform accumulation in sequence. That is, it reads all the partial sum results from the workspaces into the UB at one time (due to UB size limitations, it can only read a small portion of the partial sums from each workspace at a time), and then sequentially and cyclically accumulates the subsequent partial sums onto the first partial sum to obtain the final partial result. If the output type of Matrix C is float16, the float type result needs to be Cast to float16 on the UB, and finally, the data is written back to GM C.
4. For higher data write-out and data read efficiency, when the Vector accumulates partial sums, task partitioning is performed by element. For example, for a `16 × 16` Matrix C, assuming there are 6 Vectors, then each Vector is allocated `16 × 16/6 = 42` elements. Further considering the instruction efficiency of the Vector, a minimum partitioning block is set, and the number of elements is `256 / sizeof(ElementAccumulator)`. If `ElementAccumulator` is a 32-bit type, then at least 64 elements are partitioned per core. The `16 × 16` Matrix C is divided into 4 parts, which are processed by 4 Vectors, leaving the other 2 Vectors idle. This partitioning treats Matrix C as a continuous one-dimensional array, where each Vector processes a continuous segment of it, so that the data read and written out by the Vector is a continuous segment, resulting in higher instruction efficiency. This also results in this template being inapplicable to scenarios where Matrix C is non-contiguous, for example, where the Shape of Matrix C is 16 × 16 and the Stride of each row is 17.
5. The MultiCoreSplitkMatmul template incorporates existing CommonMatmul optimizations, including [Preload, ShuffleK, Padding, and specialized read optimizations](./CommonMatmul.md).

## 2. Application Scenarios

1. When Matrix C is small and K is large, and the cores are not fully utilized.
2. Requires Matrix C to be continuously stored on GM.
